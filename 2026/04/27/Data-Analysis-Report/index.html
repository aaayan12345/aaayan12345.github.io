\section{Database Setup}

\subsection{Environment}

\begin{itemize}
\item \textbf{Database}: MySQL 9.6.0 (installed via Homebrew on macOS)
\item \textbf{Dataset}: IBM Telco Customer Churn
(\url{https://github.com/databricks-industry-solutions/survival-analysis})
\item \textbf{LLM used}: Doubao (ByteDance) — \texttt{doubao-pro-32k}
\end{itemize}

\subsection{Database Schema}

\begin{lstlisting}[caption={DDL: create database and table}]
CREATE DATABASE telco_churn
CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

USE telco_churn;

CREATE TABLE customers (
customerID VARCHAR(20) PRIMARY KEY,
gender VARCHAR(10),
seniorCitizen TINYINT(1), – 1 = senior, 0 = non-senior
partner VARCHAR(5), – ‘Yes’ / ‘No’
dependents VARCHAR(5),
tenure INT, – months with company
phoneService VARCHAR(5),
multipleLines VARCHAR(30),
internetService VARCHAR(20), – ‘DSL’ / ‘Fiber optic’ / ‘No’
onlineSecurity VARCHAR(30),
onlineBackup VARCHAR(30),
deviceProtection VARCHAR(30),
techSupport VARCHAR(30),
streamingTV VARCHAR(30),
streamingMovies VARCHAR(30),
contract VARCHAR(20), – ‘Month-to-month’/‘One year’/‘Two year’
paperlessBilling VARCHAR(5),
paymentMethod VARCHAR(40),
monthlyCharges DECIMAL(8,2),
totalCharges DECIMAL(10,2),
churn VARCHAR(5) – ‘Yes’ / ‘No’
);
\end{lstlisting}

\section{Analytical Queries and Results}

The following queries were designed to explore churn patterns in the dataset.
Each query was first posed as a natural-language question to the LLM, and the
generated SQL was executed against the database.

\subsection{Query 1: Churn Rate by Contract Type}

\textbf{Question:} What is the churn rate for each contract type?

\begin{lstlisting}[caption={Churn rate by contract type}]
SELECT contract,
COUNT() AS total,
SUM(churn = ‘Yes’) AS churned,
ROUND(SUM(churn=’Yes’) * 100.0 / COUNT(), 2) AS churn_rate_pct
FROM customers
GROUP BY contract
ORDER BY churn_rate_pct DESC;
\end{lstlisting}

\begin{table}[H]
\centering
\caption{Churn rate by contract type}
\begin{tabular}{lccc}
\toprule
Contract & Total & Churned & Churn Rate \
\midrule
Two year & 223 & 71 & 31.84% \
Month-to-month & 530 & 149 & 28.11% \
One year & 247 & 64 & 25.91% \
\bottomrule
\end{tabular}
\end{table}

\subsection{Query 2: Average Monthly Charges by Internet Service and Churn}

\textbf{Question:} What is the average monthly charge for each internet service
type, broken down by churn status?

\section{LLM Failure Case Analysis}

The following three cases document instances where the LLM (Doubao) generated
incorrect SQL. For each case, we provide: the database schema, the user
question, the LLM-generated wrong SQL, the correct SQL, and an analysis of
the failure reason.

The prompt template used for all cases:

\begin{lstlisting}[language={},caption={LLM prompt template}]
You are a smart agent responsible for generating correct SQL statements
based on the following information:

  • Database schema: table names, fields, and their types.
  • User question: a natural language query.

Generate a single SQL statement that answers the question.
Return the result as JSON: {“sql”: ““}

Database Schema:

CREATE TABLE customers (
customerID VARCHAR(20) PRIMARY KEY,
gender VARCHAR(10), seniorCitizen TINYINT(1),
partner VARCHAR(5), dependents VARCHAR(5),
tenure INT, phoneService VARCHAR(5),
multipleLines VARCHAR(30), internetService VARCHAR(20),
onlineSecurity VARCHAR(30), onlineBackup VARCHAR(30),
deviceProtection VARCHAR(30), techSupport VARCHAR(30),
streamingTV VARCHAR(30), streamingMovies VARCHAR(30),
contract VARCHAR(20), paperlessBilling VARCHAR(5),
paymentMethod VARCHAR(40), monthlyCharges DECIMAL(8,2),
totalCharges DECIMAL(10,2), churn VARCHAR(5)
);

User Question:

\end{lstlisting}

%% ── Failure Case 1 ───────────────────────────────────────────────────────────
\subsection{Failure Case 1: Boolean Comparison with String Column}

\textbf{User Question:}
\begin{quote}
\textit{Find the churn rate for customers who have all three add-on services:
online security, online backup, and tech support.}
\end{quote}

\textbf{LLM-Generated Wrong SQL:}
\begin{lstlisting}[caption={\textcolor{wrongred}{Wrong SQL — LLM output}}]
SELECT
COUNT() AS total_customers,
SUM(CASE WHEN churn = 1 THEN 1 ELSE 0 END) AS churned_customers,
ROUND(SUM(CASE WHEN churn = 1 THEN 1 ELSE 0 END) * 100.0
/ COUNT(), 2) AS churn_rate_pct
FROM customers
WHERE onlineSecurity = 1
AND onlineBackup = 1
AND techSupport = 1;
\end{lstlisting}

\textbf{Execution Result of Wrong SQL:}
\begin{itemize}
\item Returns 0 rows / all zeros — no customers match the condition.
\end{itemize}

\textbf{Correct SQL:}
\begin{lstlisting}[caption={\textcolor{correctgreen}{Correct SQL}}]
SELECT
COUNT() AS total_customers,
SUM(churn = ‘Yes’) AS churned_customers,
ROUND(SUM(churn = ‘Yes’) * 100.0 / COUNT(), 2) AS churn_rate_pct
FROM customers
WHERE onlineSecurity = ‘Yes’
AND onlineBackup = ‘Yes’
AND techSupport = ‘Yes’;
\end{lstlisting}

\textbf{Correct Result:}
\begin{table}[H]
\centering
\begin{tabular}{ccc}
\toprule
Total Customers & Churned & Churn Rate \
\midrule
44 & 11 & 25.00% \
\bottomrule
\end{tabular}
\end{table}

\textbf{Failure Analysis:}

The LLM incorrectly assumed that boolean-like columns (\texttt{onlineSecurity},
\texttt{onlineBackup}, \texttt{techSupport}, \texttt{churn}) are stored as
integers (0/1). In reality, the schema defines them as \texttt{VARCHAR} with
values \texttt{‘Yes’} / \texttt{‘No’}. This is a \textbf{schema misinterpretation}
failure: the LLM inferred data types from column semantics rather than from the
actual DDL. The condition \texttt{onlineSecurity = 1} evaluates to false for all
rows because MySQL performs a string-to-integer cast that yields 0 for
non-numeric strings like \texttt{‘Yes’}.

%% ── Failure Case 2 ───────────────────────────────────────────────────────────
\subsection{Failure Case 2: Ranking by Churn Rate vs.\ Total Count}

\textbf{User Question:}
\begin{quote}
\textit{Rank all payment methods by their churn rate from highest to lowest,
and show the total number of customers and number of churned customers for each.}
\end{quote}

\textbf{LLM-Generated Wrong SQL:}
\begin{lstlisting}[caption={\textcolor{wrongred}{Wrong SQL — LLM output}}]
SELECT paymentMethod,
COUNT() AS total_customers,
SUM(CASE WHEN churn = ‘Yes’ THEN 1 ELSE 0 END) AS churned_customers,
RANK() OVER (ORDER BY COUNT() DESC) AS churn_rank
FROM customers
GROUP BY paymentMethod
ORDER BY churn_rank;
\end{lstlisting}

\textbf{Execution Result of Wrong SQL:}
\begin{table}[H]
\centering
\begin{tabular}{lccc}
\toprule
Payment Method & Total & Churned & Rank \
\midrule
Bank transfer (automatic) & 255 & 71 & 1 \
Credit card (automatic) & 252 & 78 & 2 \
Electronic check & 246 & 55 & 3 \
Mailed check & 247 & 80 & 4 \
\bottomrule
\end{tabular}
\end{table}

The ranking is by \emph{total customer count}, not by \emph{churn rate}.
Mailed check has the highest churn rate (32.39%) but is ranked last.

\textbf{Correct SQL:}
\begin{lstlisting}[caption={\textcolor{correctgreen}{Correct SQL}}]
SELECT paymentMethod,
COUNT() AS total_customers,
SUM(churn = ‘Yes’) AS churned_customers,
ROUND(SUM(churn=’Yes’) * 100.0 / COUNT(), 2) AS churn_rate_pct,
RANK() OVER (
ORDER BY SUM(churn = ‘Yes’) * 1.0 / COUNT(*) DESC
) AS churn_rank
FROM customers
GROUP BY paymentMethod
ORDER BY churn_rank;
\end{lstlisting}

\textbf{Correct Result:}
\begin{table}[H]
\centering
\begin{tabular}{lcccc}
\toprule
Payment Method & Total & Churned & Churn Rate & Rank \
\midrule
Mailed check & 247 & 80 & 32.39% & 1 \
Credit card (automatic) & 252 & 78 & 30.95% & 2 \
Bank transfer (automatic) & 255 & 71 & 27.84% & 3 \
Electronic check & 246 & 55 & 22.36% & 4 \
\bottomrule
\end{tabular}
\end{table}

\textbf{Failure Analysis:}

The LLM misunderstood the ranking criterion. The question asks to rank by
\textbf{churn rate} (a ratio), but the LLM used \texttt{COUNT(*)} (total
customer count) inside the \texttt{RANK() OVER} window function. This is a
\textbf{semantic ambiguity} failure: the phrase rank by churn rate'' was conflated with rank by the number of churned customers’’ or ``rank by
customer volume.’’ The LLM failed to compute the ratio
\texttt{churned / total} as the ordering key for the window function.

%% ── Failure Case 3 ───────────────────────────────────────────────────────────
\subsection{Failure Case 3: GROUP BY Ordering with CASE Buckets}

\textbf{User Question:}
\begin{quote}
\textit{Group customers into tenure buckets (1–12, 13–24, 25–36, 37–48,
49–60, 61+ months) and show the churn rate for each bucket, ordered
chronologically from shortest to longest tenure.}
\end{quote}

\textbf{LLM-Generated Wrong SQL:}
\begin{lstlisting}[caption={\textcolor{wrongred}{Wrong SQL — LLM output}}]
SELECT
CASE
WHEN tenure BETWEEN 1 AND 12 THEN ‘1-12 months’
WHEN tenure BETWEEN 13 AND 24 THEN ‘13-24 months’
WHEN tenure BETWEEN 25 AND 36 THEN ‘25-36 months’
WHEN tenure BETWEEN 37 AND 48 THEN ‘37-48 months’
WHEN tenure BETWEEN 49 AND 60 THEN ‘49-60 months’
ELSE ‘61+ months’
END AS tenure_bucket,
COUNT() AS total,
SUM(CASE WHEN churn = ‘Yes’ THEN 1 ELSE 0 END) AS churned,
ROUND(SUM(CASE WHEN churn = ‘Yes’ THEN 1 ELSE 0 END)
* 100.0 / COUNT(), 2) AS churn_rate_pct
FROM customers
GROUP BY tenure_bucket
ORDER BY tenure_bucket;
\end{lstlisting}

\textbf{Execution Result of Wrong SQL:}
\begin{table}[H]
\centering
\begin{tabular}{lccc}
\toprule
Tenure Bucket & Total & Churned & Churn Rate \
\midrule
1–12 months & 150 & 35 & 23.33% \
13–24 months & 170 & 52 & 30.59% \
25–36 months & 158 & 50 & 31.65% \
37–48 months & 160 & 33 & 20.63% \
49–60 months & 190 & 58 & 30.53% \
61+ months & 172 & 56 & 32.56% \
\bottomrule
\end{tabular}
\end{table}

The result \emph{appears} correct here because the string labels happen to sort
alphabetically in the right order. However, this is fragile: if bucket labels
were changed (e.g., \texttt{‘0-12 months’} vs \texttt{‘1-12 months’}, or
non-zero-padded numbers like \texttt{‘7-12 months’}), alphabetical ordering
would produce the wrong sequence. For example, with labels
\texttt{‘7-12’}, \texttt{‘13-24’}, \texttt{‘25-36’}, \texttt{ORDER BY
tenure_bucket} would place \texttt{‘7-12’} \emph{last} because \texttt{‘7’}
$>$ \texttt{‘3’} lexicographically.

\textbf{Correct SQL:}
\begin{lstlisting}[caption={\textcolor{correctgreen}{Correct SQL — robust ordering}}]
SELECT
CASE
WHEN tenure BETWEEN 1 AND 12 THEN ‘1-12 months’
WHEN tenure BETWEEN 13 AND 24 THEN ‘13-24 months’
WHEN tenure BETWEEN 25 AND 36 THEN ‘25-36 months’
WHEN tenure BETWEEN 37 AND 48 THEN ‘37-48 months’
WHEN tenure BETWEEN 49 AND 60 THEN ‘49-60 months’
ELSE ‘61+ months’
END AS tenure_bucket,
COUNT() AS total,
SUM(churn = ‘Yes’) AS churned,
ROUND(SUM(churn=’Yes’) * 100.0 / COUNT(), 2) AS churn_rate_pct
FROM customers
GROUP BY tenure_bucket
ORDER BY MIN(tenure); – order by the actual numeric minimum, not the label string
\end{lstlisting}

\textbf{Failure Analysis:}

The LLM used \texttt{ORDER BY tenure_bucket} (alphabetical string sort) instead
of \texttt{ORDER BY MIN(tenure)} (numeric sort on the underlying data). This is
a \textbf{ordering semantics} failure: the LLM defaulted to sorting the alias
column as a string, which coincidentally works for the current labels but is
logically incorrect. The correct approach is to order by an aggregate of the
original numeric column (\texttt{MIN(tenure)}) to guarantee chronological order
regardless of how the bucket labels are formatted.

\section{Summary: When Do LLMs Fail at SQL Generation?}

Based on the three failure cases above, we identify the following assumptions
that commonly cause LLMs to generate incorrect SQL:

\begin{enumerate}
\item \textbf{Type inference from semantics, not schema.}
LLMs often infer column data types from the column name or domain
knowledge (e.g., ``churn’’ sounds boolean, so it must be 0/1) rather
than reading the actual DDL. This leads to type mismatch errors when
the schema uses strings for categorical values.

\item \textbf{Conflating rank by X'' with rank by count of X’’.}
When a question asks to rank by a derived metric (e.g., churn
\emph{rate}), LLMs sometimes substitute a simpler aggregate (e.g.,
total count) that is easier to compute. This is a semantic
misinterpretation of ratio vs.\ absolute quantity.

\item \textbf{Defaulting to lexicographic ordering for labeled buckets.}
When \texttt{CASE} expressions produce string labels, LLMs default to
\texttt{ORDER BY alias} (string sort), which is fragile. The robust
solution is to order by an aggregate of the original numeric column
(\texttt{MIN}, \texttt{MAX}), but LLMs rarely apply this pattern
without explicit instruction.
\end{enumerate}
